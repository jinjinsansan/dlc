# Week 6: 完成させて世界に公開する

> **このWeekのゴール**: あなたのサービスが、誰でもアクセスできる URL で世界に公開される
> **所要時間**: 動画 50 分 + 教材 90 分 + 宿題 30 分
> **このWeekで身につく力**: 「自分のサービスを世界に届ける」一連の手順

---

## このWeek を始める前に

### Week 5 までの状態

あなたの手元には、**「自分の PC でだけ動くサービス」** があります。

```
今: あなたの PC のローカルでだけ動く
    URL も localhost:3000 のような、自分しかアクセスできないもの
    家族や友達に「見せて」と言われても見せられない
    ↓
今週の終わりには
    ↓
誰でもアクセスできる URL（例: yamada-portfolio.vercel.app）で世界中から見られる
    家族・友達・SNS で URL を共有できる
    検索エンジンに載る
    本物の「Web サービス」として認識される
```

### 「公開」の正体

公開とは:
1. **コードをインターネット上のサーバーにアップロード** する
2. **URL を割り当てる**
3. **ブラウザでその URL にアクセスすると、あなたのサイトが表示される**

これだけです。難しそうに聞こえますが、**今は無料で・10 分で出来ます**。

### 使うサービス

| サービス | 役割 | 料金 |
|---------|------|------|
| **GitHub** | コードの保管庫 | 無料 |
| **Vercel** | 公開サーバー | 無料（個人利用） |
| **ドメイン**（任意） | あなた専用 URL（例: my-name.com） | 年 1,000〜2,000 円 |

## レッスン 6-1: 「公開する」とは（5 分）

### Web サイトが見える仕組み（3 分で理解）

```
あなたが index.html を作る
   ↓
Vercel というサーバーに「これを置いて」と頼む
   ↓
Vercel が URL を発行（例: yamada.vercel.app）
   ↓
誰かがその URL にアクセス
   ↓
Vercel が index.html を返す
   ↓
ブラウザで表示 ✨
```

これが「Web」の本質です。**「ファイルを置いて、URL を発行してもらう」だけ**。

### 公開する前のチェック

公開前に、以下が出来ているか確認:
- [ ] サイトが PC で正しく動く
- [ ] スマホ表示も崩れない
- [ ] エラーが出ない
- [ ] AI 機能も動く

特に Week 4-5 で追加した機能が、**ローカルで完璧に動く** ことを確認してから次へ。

## レッスン 6-2: Vercel で公開する（無料）（20 分）

### ステップ全体像

```
① GitHub アカウントを作る（無料）
② プロジェクトを GitHub にアップロード（Claude にお任せ）
③ Vercel アカウントを作る（GitHub 連携で OK）
④ Vercel に GitHub プロジェクトを接続
⑤ 環境変数を設定（API キー等）
⑥ デプロイ → URL 発行 ✨
```

10〜20 分で完了します。

### ① GitHub アカウントを作る

1. [https://github.com](https://github.com) にアクセス
2. 「Sign up」で新規登録（無料）
3. ユーザー名・メール・パスワードを設定
4. メール認証を完了

### ② プロジェクトを GitHub にアップロード

Claude Code に頼みます:
```
このプロジェクトを GitHub にアップロードしたい。
私の GitHub ユーザー名: yamada-taro

【やってほしいこと】
1. .gitignore ファイルを作成（node_modules, .env.local 等を除外）
2. README.md を整備
3. GitHub に新しいリポジトリを作る手順を教えて
4. ローカルから push する手順を教えて
```

→ Claude が手順を全部教えてくれます。

#### 補足: .env.local は絶対に push しない

`.env.local` には API キーなどの **秘密情報** が入っています。これを GitHub に上げると **誰でも盗める状態** になります。
`.gitignore` に `.env.local` を必ず含めること（Claude が自動でやってくれます）。

### ③ Vercel アカウントを作る

1. [https://vercel.com](https://vercel.com) にアクセス
2. 「Sign Up」をクリック
3. **「Continue with GitHub」** を選択（GitHub アカウントで連携）
4. 個人利用 → Hobby プラン（無料）

### ④ Vercel に GitHub プロジェクトを接続

1. Vercel の Dashboard で「Add New... → Project」
2. 表示された GitHub リポジトリ一覧から **自分のプロジェクト** を選ぶ
3. 「Import」をクリック
4. プロジェクト設定画面が出る

### ⑤ 環境変数を設定

`.env.local` に書いた API キーを Vercel にも設定する必要があります。

Vercel のプロジェクト設定画面で:
1. 「Environment Variables」セクション
2. **Key** と **Value** に、`.env.local` の内容を 1 つずつコピペ:
   ```
   NEXT_PUBLIC_SUPABASE_URL = https://xxx.supabase.co
   NEXT_PUBLIC_SUPABASE_ANON_KEY = eyJhbG...
   SUPABASE_SERVICE_ROLE_KEY = eyJhbG...
   ANTHROPIC_API_KEY = sk-ant-...
   ```
3. **「Production」「Preview」「Development」全てにチェック**

### ⑥ Deploy

1. 一番下の「Deploy」ボタンをクリック
2. 1〜3 分待つ
3. 「Congratulations!」画面 + 自分の URL（例: `https://yamada-portfolio-xxx.vercel.app`）

→ **その URL をブラウザで開く** → あなたのサイトが世界中からアクセス可能に ✨

### 動作確認

1. URL をスマホで開く → サイトが表示される
2. 家族・友達に URL を送る → 「ちゃんと見えるよ」と言われる
3. AI 機能、ログイン機能も動くか確認

## レッスン 6-3: 自分のドメインをつける（15 分）

### 「.vercel.app」が嫌なら、自分専用 URL を取得

`yamada-portfolio.vercel.app` ↑ これは無料だけど Vercel の URL。
自分専用にしたいなら、ドメイン（独自 URL）を取得します。

### ドメイン取得 (年 1,000〜2,000 円)

#### おすすめドメイン取得サービス

| サービス | おすすめ度 | 価格目安 |
|---------|-----------|---------|
| お名前.com | ⭐⭐⭐ | .com で 1,500 円/年〜 |
| Cloudflare Registrar | ⭐⭐⭐⭐⭐ | 卸値で安い、信頼性高 |
| Google Domains | （現在停止中） | — |
| ムームードメイン | ⭐⭐⭐ | 価格安い、UI 日本語 |

**初心者は Cloudflare Registrar 推奨**: 安い + 設定簡単 + 安心。

### ドメイン名の決め方

```
✅ 良い例
- yamada-taro.com（自分の名前）
- ai-counselor.app（サービス名）
- 5min-recipe.com（覚えやすい）

❌ 微妙な例
- my-super-cool-amazing-website.com（長すぎ）
- 12345.xyz（怪しげ）
- my_website_2026.com（_ や数字は避ける）
```

### Vercel に独自ドメインを接続

1. Vercel プロジェクトの「Settings → Domains」
2. 「Add Domain」で取得したドメインを入力
3. 表示される **DNS 設定** をドメイン会社で設定
4. 5〜30 分待つ → ドメインが繋がる ✨

困ったら Claude に:
```
取得したドメイン yamada.com を Vercel に接続したい。
お名前.com の DNS 設定で何をすればいい?
```

## Vercel 公開手順書（完全版）

### 手順 1: GitHub の準備

```bash
# プロジェクトフォルダで（ターミナルから）
cd ~/my-apps/my-website

# Git 初期化（既に初期化済みならスキップ）
git init

# .gitignore を確認（.env.local が含まれているか）
cat .gitignore | grep .env

# 全ファイルをステージング
git add .

# コミット
git commit -m "Initial commit"
```

### 手順 2: GitHub にリポジトリを作る

1. GitHub にログイン
2. 右上の「+」→「New repository」
3. **Repository name**: `my-website` 等
4. **Public** か **Private** を選択（Private 推奨）
5. README は **追加しない**（既にあるため）
6. 「Create repository」をクリック
7. 表示されるコマンドをターミナルにコピペ:
   ```bash
   git remote add origin https://github.com/your-username/my-website.git
   git branch -M main
   git push -u origin main
   ```

### 手順 3: Vercel に接続

1. Vercel にログイン
2. Dashboard で「Add New」→「Project」
3. GitHub の自分のアカウントを認証
4. リポジトリ一覧から `my-website` を選択 → 「Import」

### 手順 4: 環境変数の設定

「Environment Variables」セクションで、以下を全て追加（`.env.local` の内容と同じ）:

```
NEXT_PUBLIC_SUPABASE_URL = https://xxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY = eyJhbG...
SUPABASE_SERVICE_ROLE_KEY = eyJhbG...
ANTHROPIC_API_KEY = sk-ant-...
NEXT_PUBLIC_ADMIN_EMAIL = your@email.com
（必要に応じて他も）
```

各変数で **Production / Preview / Development の 3 つにチェック**。

### 手順 5: Deploy

1. 「Deploy」ボタンをクリック
2. 1〜3 分待つ
3. 「Congratulations!」画面が出る
4. 「Visit」ボタンで自分のサイトを開く 🎉

### 手順 6: その後の更新

```bash
# コードを修正
# ↓
git add .
git commit -m "○○を修正"
git push
# ↓
# Vercel が自動で再デプロイ（1〜2 分）
```

または Claude に「git に変更を push して」と頼むだけ。

## ドメイン取得ガイド

### 推奨サービス比較

#### Cloudflare Registrar（一番おすすめ）
- **価格**: 卸値で最安（.com は約 $10/年）
- **更新料金**: 値上げなし
- **DNS**: 同じく Cloudflare で管理可能（高速・無料）
- **欠点**: 英語 UI、日本円決済不可（クレジットカード）

#### お名前.com（日本人に人気）
- **価格**: 初年度安い（300 円〜）が **更新時に高くなる**（3,000 円超）
- **DNS**: 自社 DNS 利用可
- **メリット**: 日本語、初心者向け
- **欠点**: 営業メールが多い、UI が古い

#### ムームードメイン
- **価格**: 1,500 円/年〜
- **メリット**: シンプル、日本語、安定
- **欠点**: 機能少なめ

### ドメインの選び方

```
✅ できれば .com or .jp（信頼度高）
✅ 短く（10 文字以内が理想）
✅ 覚えやすい
✅ サービス名と一致 or 自分の名前
✅ ハイフンや数字は避ける（覚えにくい）

❌ 流行りの拡張子（.xyz, .info）は避ける
❌ ブランド名と紛らわしい（appleの.com 等）
❌ 商標登録されている名前
```

## トラブルシューティング

### Q1. Deploy したらビルドエラーが出ました

**A**: エラー画面のログを Claude に渡す:
```
Vercel デプロイで以下のエラーが出た。直して:
[エラー文をコピペ]
```
→ ほとんどは「環境変数が設定されていない」「ローカルにしかないファイルを参照している」等。

### Q2. ローカルでは動くのに、本番で AI 機能が動かない

**A**: 環境変数の設定漏れ。Vercel の「Environment Variables」を確認:
- 全ての .env.local の値が入っているか
- 「Production」にもチェックが入っているか
- 値の前後に余分なスペースがないか

設定後、再デプロイ（Vercel の「Redeploy」ボタン）。

### Q3. 独自ドメインが繋がらない

**A**: DNS の伝播に時間がかかります（最大 48 時間、通常は 30 分〜2 時間）。
- Vercel の Domains 画面で「Valid Configuration」になっているか確認
- ならない場合は、ドメイン会社の DNS 設定が間違っている可能性
- Claude に DNS 設定を確認してもらう

### Q4. URL を友達に送ったら「アクセスできない」と言われた

**A**:
- URL のタイポを確認
- HTTPS（`https://`）になっているか確認
- 自分のスマホで別の Wi-Fi（携帯回線）から開いて確認

### Q5. Supabase の RLS（Row Level Security）でエラーが出る

**A**: 本番ではセキュリティが厳しくなります。Claude に頼む:
```
Supabase の RLS 設定を本番用にして。
○○テーブルは認証済みユーザーが自分のデータだけ読み書き可能、
管理者は全て可能、にして。
```

### Q6. `.env.local` の内容が GitHub に push されてしまった!

## コラム: 「URL を持つ」という新しい体験

私が初めて自分のサービスを公開した日のことを覚えています。

`http://example.vercel.app` を妻に送って「これ、見て」と言ったら、妻のスマホに表示された。
それまで「自分の頭の中・PC の中」にしか無かったものが、**世界中からアクセスできる場所** に存在するようになった。

この瞬間、世界の見え方が変わりました。

> **「自分は、世界に何かを届けられる人間だ」**

これが、Web 個人開発の最も尊い体験だと思います。

Week 6 を終えたあなたは、もう **「サービスを世界に届けられる人」** です。
おめでとう、本当に。

次の Week では、これに **「お金を受け取る仕組み」** を入れて、**ビジネス** にしていきます。
