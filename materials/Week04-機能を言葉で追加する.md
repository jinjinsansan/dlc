# Week 4: 機能を言葉で追加する

> **このWeekのゴール**: 「ただ見るだけのサイト」から「動くアプリ」へ。データ保存・ログイン・マイページが動く本物の Web アプリにする
> **所要時間**: 動画 50 分 + 教材 90 分 + 宿題 30 分
> **このWeekで身につく力**: 「機能を日本語で頼んで実装する力」。これが「Web サービス開発者」の核心スキル

---

## このWeek を始める前に

### Web サイト → Web アプリ への進化

```
Web サイト（Week 1〜3）:
  ・見るだけ
  ・お問い合わせを送ってもデータは保存されない（フリ）
  ・誰が来ても同じ画面

Web アプリ（Week 4〜）:
  ・操作できる
  ・データが本当に保存される（消えない）
  ・ログインしたユーザー専用の画面が出せる
```

この差は、**個人の趣味サイト** と **本物のサービス** の境界線です。
Week 4 でこの境界を越えます。

### 必要な新しい知識（5 分でわかる）

これから登場する 3 つの言葉を、ざっくり理解しておきましょう。

| 用語 | これ何？ | 例えるなら |
|------|---------|----------|
| **データベース** | データを保存しておく場所 | 巨大なエクセルファイル |
| **認証** | 「この人は誰か」を確認する仕組み | マンションのオートロック |
| **API** | アプリ同士がやり取りする仕組み | 飲食店の注文票 |

これらを **「Claude にお任せ」** で実装します。仕組みを完璧に理解する必要はありません。

## レッスン 4-1: 「動くアプリ」とは（10 分）

### 例 ①: お問い合わせフォームの違い

#### サイト（フリだけ）
```
1. ユーザーが名前・メール・本文を入力
2. 「送信」ボタンを押す
3. 「ありがとうございました」と表示
4. でも、実は何も送られていない（データは消える）
```

#### アプリ（本物）
```
1. ユーザーが名前・メール・本文を入力
2. 「送信」ボタンを押す
3. データがデータベースに保存される
4. 管理者にメール通知が届く
5. 管理者は管理画面で問い合わせ一覧を見られる
```

この差を、Week 4 で実装できるようになります。

### 例 ②: 会員制コンテンツ

```
ログインしていない人 → トップページしか見られない
ログインした人 → マイページ・限定コンテンツが見られる
管理者 → 全ユーザーの情報・売上等が見られる
```

これが Week 7 の決済とも繋がる重要な機能です。

## レッスン 4-2: お問い合わせフォームをちゃんと動くようにする（20 分）

### Supabase（データベースサービス）の紹介

データを保存するために **Supabase** という無料サービスを使います。
**理由**:
- 個人利用は完全無料
- 設定が簡単（Claude に任せられる）
- データベース・認証・ファイル保存が全部入り
- 後の Week でも同じものを使い続ける

### 準備: Supabase アカウントを作る

1. [https://supabase.com](https://supabase.com) を開く
2. 「Start your project」をクリック
3. GitHub アカウント or メールで登録（無料）
4. 「New project」で新しいプロジェクトを作成
   - プロジェクト名: 自分のサイト名（例: `my-website`）
   - データベースパスワード: 強めのパスワードを設定（メモ必須）
   - リージョン: Tokyo を選択
5. 1〜2 分待つと、プロジェクトが立ち上がる
6. 左サイドバーの **「Project Settings」 → 「API」** を開く
7. 以下の 2 つをコピーしてメモ:
   - **Project URL**（`https://xxx.supabase.co`）
   - **anon key**（長い文字列）

### Claude Code に伝える

```
Supabase をこのプロジェクトに導入して。

【Supabase 情報】
- Project URL: https://xxx.supabase.co
- anon key: eyJhbGc...

【やってほしいこと】
1. Supabase クライアントの設定
2. お問い合わせデータを保存するテーブルを Supabase に作る SQL を生成
3. お問い合わせフォームから送信したら、Supabase に保存されるようにする
4. 送信後「ありがとうございました」のメッセージ
```

→ Claude が必要なファイル変更と、Supabase の管理画面で実行する SQL を出してくれます。

### Supabase 管理画面で SQL を実行

1. Supabase の画面で **左サイドバーの「SQL Editor」** を開く
2. Claude が出した SQL をコピペ
3. 「Run」ボタンを押す

→ テーブル（データの入れ物）が作成されます。

### 動作確認

サイトのお問い合わせフォームから送信 → Supabase の「Table Editor」で見ると **本当にデータが入っている** ✨

これがあなたの **「サイトをアプリにした」初体験** です。

## レッスン 4-3: ログイン機能をつける（20 分）

### Supabase Auth の紹介

Supabase には **「ユーザー認証機能」** が標準で入っています。これを使えば、ログイン・会員登録が **数十分で実装** できます。

### Claude に頼む

```
ユーザー認証機能を実装して。

【機能】
- 会員登録ページ（メール+パスワード）
- ログインページ
- ログアウト機能
- ログイン中のユーザーは右上に名前を表示
- ログインしていないユーザーは「ログイン」ボタンを表示

【使うサービス】
Supabase Auth
```

→ Claude が認証関連のページとロジックを全部作ってくれます。

### Supabase 側の設定

Supabase の管理画面で以下を確認:
1. 左サイドバー「Authentication」を開く
2. 「Providers」→ 「Email」が ON になっていることを確認
3. 「Settings」で「Confirm email」を OFF にしておく（テスト中は楽）
   - 本番では ON 推奨。Week 6 で切り替える

### 動作確認

1. 自分のサイトで会員登録 → Supabase「Authentication」→「Users」に自分が追加されている ✨
2. ログイン → 右上に名前表示
3. ログアウト → 元に戻る

## レッスン 4-5: 管理画面を作る（15 分）

### 管理画面とは

**自分（運営者）だけ** がアクセスできる、サービスの裏側を管理するページ。

### Claude に頼む

```
管理画面を作って。

【パス】
/admin

【アクセス制御】
特定のメールアドレスでログインしている人だけアクセス可能
（私のメール: your@email.com）
他の人は「アクセス権限がありません」と表示

【表示内容】
1. 全ユーザー一覧（メール・登録日）
2. お問い合わせ一覧（日付の新しい順）
3. お問い合わせの詳細表示・対応済みフラグ
4. ユーザー数・問い合わせ数の統計
```

→ かっこいい管理画面が出来ます。

### 管理者判定の仕組み

`.env.local` ファイルに `ADMIN_EMAIL=your@email.com` を設定。
Claude は **環境変数（.env.local）** で管理者を判定するように作ってくれます。これにより、コードを公開しても管理者メールが漏れません。

## レッスン 4-6: エラーが出た時の対処法（10 分）

Week 4 から、エラーに遭遇する頻度が上がります。**でも、対処法は超シンプル** です。

### 対処の 3 ステップ

```
Step 1: ブラウザの開発者ツールで赤い文字（エラー）を確認
       ↓
Step 2: その赤い文字を全部コピー
       ↓
Step 3: Claude に「このエラーが出た。直して」と貼り付ける
```

これだけ。9 割のエラーはこれで解決します。

### 開発者ツールの開き方

| ブラウザ | ショートカット |
|---------|---------------|
| Chrome (Mac) | `⌘ + Option + I` |
| Chrome (Win) | `F12` または `Ctrl + Shift + I` |
| Safari | `⌘ + Option + I`（事前に開発メニュー有効化） |

開いたら **「Console」タブ** を見ます。赤い文字 = エラーです。

### スクショで渡す方法

開発者ツールのスクリーンショットを撮って、Claude にドラッグ&ドロップ:
```
このエラーが出ています。直して。
[画像]
```

→ Claude が画像を解析して原因と修正コードを返してくれます。

### よくあるエラー TOP3

#### ① 「supabase is not defined」
原因: Supabase クライアントの import 忘れ
解決: 「Supabase クライアントを正しく import して」と Claude に頼む

#### ② 「Hydration error」
原因: サーバーとクライアントの表示内容が違う
解決: エラー文を貼って「Hydration エラーを直して」

#### ③ 「Cannot read property 'xxx' of undefined」
原因: データがまだ無い時に表示しようとした
解決: エラー文を貼って「ローディング処理を入れて」

## 「エラーが出たらこうする」3 ステップガイド

### Step 1: 落ち着く

エラーは **「PC が壊れた」のではなく「AI に頼めば直る、よくあるやつ」** です。
ほとんどのエラーは、過去の受講者全員が一度はぶつかったもの。

### Step 2: エラー情報を集める

#### A. ブラウザのエラー
1. 右クリック → 「検証」→「Console」タブ

1. ターミナル画面の赤い文字をマウスで選択 → コピー

#### C. 画面の見た目がおかしい
1. スクリーンショットを撮る（Mac: `⌘+Shift+4` / Win: `Win+Shift+S`）

### Step 3: Claude に渡す

```
エラーが出ました。直してください。

【症状】
○○ボタンを押したら△△が表示されない

【ブラウザのコンソールエラー】
（コピペ）

【スクリーンショット】
（画像をドラッグ&ドロップ）
```

→ ほぼ 100% 解決します。

## トラブルシューティング

### Q1. Supabase の画面が英語で分からない

**A**: 主な画面は以下:
- **Table Editor**: 保存されたデータを見る
- **SQL Editor**: SQL を実行する
- **Authentication > Users**: 登録ユーザーを見る
- **Project Settings > API**: URL と Key を取得
他は基本触りません。

### Q2. 「Invalid API key」というエラーが出ます

**A**: `.env.local` の Key の貼り付けミスが多いです:
- 改行や余分なスペースが入っていないか確認
- key 全体（長い文字列）を再コピペ
- Claude に「.env.local の SUPABASE_KEY が無効と言われる。確認して」と頼む

### Q3. メールが届かないので確認できません

**A**: Supabase の設定で:
- 「Authentication > Settings > Confirm email」を一時的に **OFF** にしておく
- これでメール確認なしでログイン可能（テスト中のみ）

### Q4. ログインしてもすぐログアウトされてしまいます

**A**: Cookie の設定の問題:
```
ログインしてもすぐログアウトされる。Cookie の設定を確認して直して
```
→ Claude が修正します。

### Q5. ローカルでは動くのに、本番で動きません

**A**: これは Week 6 で詳しく扱います。今は気にしないで OK。

### Q6. テーブルを間違って作ってしまった

**A**: Supabase の Table Editor で削除 → SQL を作り直して再実行。データを失っても OK な段階なので恐れず。

## コラム: 「DB（データベース）って怖くない」の話

Week 4 までの受講者の感想で、最も多いのが:

> 「データベースってもっと難しいものだと思ってた。実は『データを保存する場所』というだけだった」

その通りです。

**データベース** = 巨大なエクセルファイル。
**SQL** = エクセルに「○○の行を取ってきて」とお願いする命令。
**API** = アプリ同士の電話。

これらは **「中の仕組みを完璧に理解する」必要がない** のが、Claude Code 時代の最大の革命です。
やりたいことを日本語で頼めば、AI が裏側を全部やってくれます。

私が 5 年前に競馬予想 AI を作り始めた時、データベース設計に **3 ヶ月** かかりました。
今は同じことが **1 日** で出来ます。

これが、**今 AI 個人開発を始める人が「圧倒的に得をする時代」** の正体です。
